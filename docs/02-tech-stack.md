# 02. 技術スタックと選定理由

このドキュメントは、StacX の技術選定における判断軸と理由を記録します。職務経歴書・技術面接で説明できるレベルの粒度で記述します。

---

## 全体構成

| レイヤー | 技術 |
|---|---|
| フロントエンド | React Router v7（フレームワークモード）+ TypeScript |
| UI ライブラリ | shadcn/ui + Tailwind CSS |
| バックエンド | Hono on Cloudflare Workers |
| 型共有 | Hono RPC |
| データベース | Cloudflare D1（SQLite） |
| ORM | Drizzle ORM |
| 認証 | OIDC（Google）+ arctic + 自前セッション管理 |
| ホスティング | Cloudflare Pages + Cloudflare Workers |
| パッケージ管理 | pnpm workspace |

---

## 選定理由

### React Router v7 を選んだ理由

**判断軸**: 業務で使用している RR v7 の理解を深掘りし、Hono との思想的一貫性を保つ。

- 業務で RR v7 を使用しており、フレームワークモードの理解を進めたい
- `loader` / `action` による Web 標準準拠のデータフローが、StacX の「フォーム入力 → DB 保存」中心の動作と完璧にマッチ
- Hono と同じく Web 標準準拠の思想を持ち、両者の組み合わせで一貫性が出る
- Next.js も検討したが、Server Components の恩恵を受けるほどの複雑性は StacX に無く、選定理由として弱い

### Hono を選んだ理由

**判断軸**: StacX の体験価値「速さ」をエッジ実行で物理的に実現する。

- StacX の主要動作は「思いついた瞬間にメモを保存」であり、レスポンスの速さが UX を決定づける
- Cloudflare Workers のエッジ実行により、世界中どこからでも低レイテンシでアクセス可能
- 業務で使う Go との比較を経た上での選定（後述）
- Hono RPC によりフロントエンドと型を共有でき、別途 OpenAPI 等のスキーマ管理が不要

### なぜ Go ではなく Hono か

**判断軸**: 業務スキルの活用と個人開発の価値最大化のバランス。

- 業務で Go を使用しているため、Go スキルは業務経験で証明できる
- 個人開発では業務で触れない領域（エッジコンピューティング、TS による型共有設計）を試す方が、ポートフォリオの幅が広がる
- StacX の規模（個人〜小規模 SaaS、低 QPS、シンプルな業務ロジック）では、Go の並行処理性能等の優位性が活きない
- フロントとバックを TypeScript で統一することで、個人開発の速度が大幅に向上

**面接で語る場合の表現例**:
> 「業務で Go を使用していますが、StacX の特性を考えると Go の優位性が活きる場面が少ないと判断しました。むしろ、業務で触れていないエッジ環境と TS による型共有設計を試すことで、技術選定の判断軸を増やすことを優先しました。」

### Cloudflare D1 を選んだ理由

**判断軸**: Workers との統合性、運用負荷の低さ、コスト。

- Workers と同じ Cloudflare エコシステムで完結、低レイテンシ
- SQLite 互換でローカル開発が容易（`wrangler dev` で本番同等の挙動）
- 個人利用〜小規模 SaaS では無料枠で運用可能
- StacX の制約（書き込み頻度低、データ量少、複雑なクエリ不要）に完全に適合
- SaaS 化時、テナントごとに DB を分ける戦略も取れる（書き込み性能の制約を回避可能）

**懸念点と対策**:
- 書き込み性能の上限（〜数十 req/s）→ 個人ツールでは問題なし、SaaS 化時は分割対応
- 大規模分析クエリ非対応 → StacX の用途で不要

### Drizzle ORM を選んだ理由

**判断軸**: 型安全性、SQL ライクな記述、エッジ環境対応。

- TypeScript の型推論が極めて強力（スキーマからクエリ結果まで型が通る）
- SQL に近い API で、業務で使う sqlc の思想と近い
- D1 / Cloudflare Workers の公式サポートあり
- マイグレーションツール（drizzle-kit）も同梱で完結

### OIDC（自前実装）を選んだ理由

**判断軸**: 認証フローの理解、SaaS 化への拡張性、依存の最小化。

- 認証フローを自分で実装することで OAuth2/OIDC の仕組みを深く理解
- `arctic` ライブラリで多数の IdP に対応可能、SaaS 化時の拡張が容易
- Auth0 等のマネージドサービス依存を避け、コスト・移行リスクを最小化
- セッション管理を D1 で行うことで、エッジ環境での認証設計を学習

詳細: `docs/05-auth.md`

### pnpm workspace を選んだ理由

**判断軸**: モノレポ管理のシンプルさ、Turborepo との比較。

- フロント / バック / 共有コードを 1 リポジトリで管理し、型共有を容易に
- Turborepo も検討したが、StacX の規模では過剰
- pnpm workspace は標準機能のみで十分、学習コストが低い

## 検討したが採用しなかった技術

| 技術 | 検討した理由 | 不採用の理由 |
|---|---|---|
| Next.js | エコシステムが大きい | Server Components の恩恵が StacX には薄い、RR v7 業務経験を活かせない |
| Go + sqlc | 業務スキルそのまま、王道 | 業務で証明済み、個人開発では別領域を試したい、StacX 規模ではオーバースペック |
| HonoX | Hono だけで完結、尖った構成 | RR v7 業務スキルを活かせない、情報少、UI 構築コスト高 |
| Prisma | 王道 ORM | エッジ対応が後発、Drizzle の方が D1 と相性良い |
| Auth.js | RR v7 でも使える | OIDC を自前実装することで学習価値が高い |
| Auth0 / Clerk | 実装が楽 | コスト・依存リスク、認証フロー理解の機会損失 |
| PostgreSQL（Neon） | 高機能、汎用 | エッジ実行の利点を活かしきれない、D1 で十分 |
