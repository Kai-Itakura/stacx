## プロジェクト概要

**StacX** は、フルスタックエンジニアが日々の業務での学び・成果・技術的判断を「1 分メモ」として素早く蓄積し、転職活動時に職務経歴書として出力できる個人向けアプリケーションです。

### コア価値
業務中の「あ、これ経歴書に使えるかも」という瞬間を逃さず、後から経歴書として再利用できる形に昇華させる。

### ターゲットユーザー
- 第一フェーズ: 開発者本人（自社開発企業への転職を目指すフルスタックエンジニア）
- 第二フェーズ: 同様の課題を持つエンジニア向け SaaS

詳細: `docs/01-product-vision.md`

## 技術スタック

### フロントエンド
- **React Router v7**（フレームワークモード）
- **TypeScript**
- **shadcn/ui** + **Tailwind CSS**

### バックエンド
- **Hono** on **Cloudflare Workers**
- **Hono RPC**（フロントエンドとの型共有）

### データベース
- **Cloudflare D1**（SQLite）
- **Drizzle ORM**

### 認証
- **OIDC**（Google IdP）
- **arctic** ライブラリ + 自前セッション管理（D1 保存）

### インフラ・ツール
- **Cloudflare Pages**（フロントエンドホスティング）
- **Cloudflare Workers**（API ホスティング）
- **pnpm workspace**（モノレポ管理）

## 設計判断の根拠（重要）

エージェントがコードを書く際、以下の判断基準を尊重してください。

### なぜ Hono + RR v7 か
- StacX の体験価値は「速さ」。エッジ実行（Workers）で物理的に実現
- 両者とも Web 標準準拠で思想が揃う
- TypeScript で統一、Hono RPC で型を共有

### なぜ Go ではなく Hono か
- StacX の規模・複雑度では Go の優位性が活きない
- 業務で Go を使用済みのため、個人開発ではエッジ環境を試す意義が大きい
- フロントとバックを同一言語にすることで開発速度が向上

### なぜ D1 か
- Workers との統合で低レイテンシ
- 個人利用〜小規模 SaaS に最適な性能・コスト
- SQLite ベースでローカル開発が容易

### なぜ OIDC（自前実装）か
- 認証フローを自分で実装することで仕組みを深く理解
- SaaS 化時のマルチ IdP 対応への拡張性
- マネージドサービス依存を避けつつセキュアに実装

詳細: `docs/02-tech-stack.md`

## コーディング規約

### 全般
- TypeScript の strict モードを有効化
- 型は可能な限り明示（`any` は禁止、`unknown` を活用）
- 関数・変数名は意図が伝わる命名（略語を避ける）

### フロントエンド
- React Router v7 の `loader` / `action` を活用したデータフロー
- クライアントサイド状態は最小限、URL 状態を優先
- shadcn/ui コンポーネントを基本とし、カスタムは最小限に
- フォームは `Form` コンポーネント + `action` を基本とする

### バックエンド
- Hono RPC を活用してフロントに型を配布
- Drizzle のクエリビルダーを使用、生 SQL は最小限
- バリデーションは Zod で統一
- エラーは型として表現（throw より Result 型を推奨）

### 認証
- セッショントークンは httpOnly Cookie で管理
- CSRF 対策は SameSite=Lax を基本
- すべての保護されたルートで認証ミドルウェアを通す

詳細: `docs/06-development.md`

## 開発フェーズ

### Phase 1: MVP（個人利用）
- 4 画面すべてを実装
- 認証は Google OIDC のみ
- 自分専用のシングルテナント想定

### Phase 2: SaaS 化
- マルチテナント対応
- 複数 IdP 対応
- 課金フロー追加

現在は **Phase 1**。マルチテナント前提の設計を意識しつつ、過剰実装は避ける。

## エージェント向け作業ガイドライン

1. **不明な点があれば必ず質問する**。推測でコードを書かない
2. **設計判断が必要な変更は事前にユーザーに確認**
3. **既存のコーディング規約・ディレクトリ構成を尊重**
4. **大きな変更は小さなコミット単位に分割**
5. **新しいライブラリ導入は提案ベース**、勝手に追加しない
6. **テストコードは要求があった時のみ作成**（過剰実装を避ける）
